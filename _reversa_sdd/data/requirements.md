## Visão Geral

A feature Data é a camada de dados do portfólio — quatro arquivos JSON estáticos que alimentam os componentes de conteúdo. Não há banco de dados, não há API, não há fetch em runtime: os dados são importados diretamente pelos componentes React em tempo de build e bundled no JavaScript final. O autor edita estes arquivos diretamente para atualizar seu histórico profissional e lista de projetos.

## Responsabilidades

- Armazenar o histórico profissional em `jobs.json` (5 entradas) 🟢
- Armazenar o portfólio de projetos em `toshi-projects.json` (19 entradas) 🟢
- Armazenar o registro de tecnologias em `stacks.json` (ícones e nomes) 🟢
- Armazenar o registro de ferramentas em `swtools.json` (ícones e nomes) 🟢
- Garantir que projetos e jobs sejam exibidos com o mais recente no topo (inversão no componente) 🟢

## Regras de Negócio

- **BR-04** — Dados armazenados em ordem cronológica (mais antigo primeiro). Os componentes `Jobs.tsx` e `Projects.tsx` aplicam `_.reverse([...data])` no render para exibir o mais recente no topo. Os dados originais nunca são mutados. 🟢
- **BR-05** — As duas entradas de educação (UNINOVE 2008–2010, FATEC 2010–2013) são JSX hardcoded em `Jobs.tsx` e **não** estão em `jobs.json`. `jobs.json` contém apenas experiências profissionais. 🟢
- **BR-07** — O campo `label` em `toshi-projects.json` deveria ser único. 5 entradas compartilhavam `label: "requirement"` (AXA requirement, NBWF, local-file-uploader, web-template-string, underwriting improvement). **Corrigir na reimplementação** com labels únicos aprovados pelo autor e migração do accordion para usar `id` como chave. 🟢 [Decisão confirmada — Pergunta 2]
- O campo `ongoing: true` em jobs e projetos indica trabalho em andamento — sem data de fim, exibido com "Present" no componente. 🟢
- `stacks.json` e `swtools.json` têm schema idêntico (id, name, class, img) mas representam categorias diferentes (tecnologias vs ferramentas). 🟢
