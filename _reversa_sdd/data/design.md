## Interface

Esta feature não expõe funções ou componentes — é uma camada de dados pura. A "interface" são os schemas dos arquivos JSON.

### Schema: `jobs.json`

```ts
interface JobEntry {
  id: string          // identificador único (ex: 'axadev')
  company: string     // nome da empresa
  role: string        // cargo
  start: string       // 'YYYY-MM-DD'
  end: string | null  // 'YYYY-MM-DD' | null quando ongoing=true
  ongoing: boolean    // true se ainda trabalhando
  technologies: string[]  // nomes de tecnologias (loose ref a stacks.json)
  description: string // chave i18n ou texto direto
}
```

**Entradas (5):** AXA (dev), AXA (designer), Infosys, Freelance, Agência SP
**Ordem no arquivo:** mais antigo primeiro → `_.reverse()` no componente para exibição

### Schema: `toshi-projects.json`

```ts
interface ProjectEntry {
  id: string          // identificador numérico string (ex: '1')
  label: string       // ⚠️ intencionalmente único mas 5 compartilham 'requirement'
  title: string       // título do projeto
  description: string // chave i18n ou texto direto
  stack: string[]     // nomes de tecnologias (loose ref a stacks.json)
  link: string | null // URL do projeto em produção
  github: string | null // URL do repositório
  category: string    // categoria do projeto
  from: string        // 'YYYY-MM-DD'
  to: string | null   // 'YYYY-MM-DD' | null quando ongoing=true
  ongoing: boolean
  role: string        // papel do autor no projeto
  highlights: string[]  // pontos de destaque
  images: string[] | null  // filenames em /public/img/ (campo possivelmente não renderizado)
}
```

**Entradas (19):** 19 projetos em ordem cronológica (mais antigo primeiro)
**Bug BR-07:** 5 entradas com `label: "requirement"` — AXA requirement design, NBWF, local-file-uploader, web-template-string, underwriting improvement

### Schema: `stacks.json` / `swtools.json`

```ts
interface StackEntry {
  id: string    // identificador único
  name: string  // nome de exibição (ex: 'React')
  class: string // classe CSS para ícone (ex: 'devicon-react-original')
  img: string   // filename do ícone em /public/img/stacks/
}
```

**stacks.json:** ~20 tecnologias (React, Next.js, TypeScript, Java, etc.)
**swtools.json:** ~15 ferramentas (Figma, VS Code, Git, etc.)

## Fluxo Principal — Consumo dos Dados

```
Componente importa dados:
  import data from 'src/data/jobs.json'
  // ↑ ES module import — bundled no JS pelo webpack do Next.js

Componente inverte ordem:
  const sorted = _.reverse([...data])
  // ↑ spread para não mutar o array original
  // ↑ _.reverse é in-place, por isso o spread é necessário

Componente renderiza:
  sorted.map(entry => <TimelineItem key={entry.id} {...entry} />)
```

**Não há fetch em runtime.** Os dados estão no bundle JS gerado pelo `next build` — disponíveis imediatamente após hidratação, sem waterfall de rede.

## Estrutura de Dados — Relacionamentos

```
jobs.json
  └── technologies[] → string[]  (loose ref — não há FK para stacks.json)

toshi-projects.json
  └── stack[] → string[]  (loose ref — não há FK para stacks.json)
  └── images[] → string[] (filenames em /public/img/ — campo sem uso confirmado em UI)

stacks.json
  └── img → string (filename em /public/img/stacks/)

swtools.json
  └── img → string (filename em /public/img/stacks/)
```

Não há joins, lookups ou relacionamentos enforced. Os componentes usam os strings de `technologies[]` e `stack[]` diretamente para exibição — não fazem lookup em `stacks.json` para obter ícones.

## Decisões de Design Identificadas

| Decisão | Evidência no código | Confiança |
|---------|---------------------|-----------|
| JSON plano como data layer (sem banco, sem API) | `src/data/*.json` — arquivos estáticos editáveis | 🟢 |

| `images[]` em projects — campo presente no schema mas sem uso confirmado na UI | `toshi-projects.json` — campo presente, sem referência em `Projects.tsx` | 🟡 |

## Estado Interno

Nenhum. Os arquivos JSON são read-only em runtime. Nenhum componente modifica os dados — toda mutação é feita pelo autor via edição direta dos arquivos e novo deploy.

## Observabilidade

Nenhum evento GA ou log associado à camada de dados. Erros de schema (ex: campo obrigatório ausente) causam erros de runtime em React (TypeError no render) — sem tratamento de erro implementado.

## Riscos e Lacunas

- 🔴 **Bug BR-07 (labels duplicados):** corrigir exige: (1) atualizar os 5 registros em `toshi-projects.json` com labels únicos, (2) migrar o state de `expanded[]` de `label` para `id` em `Projects.tsx`. Impacto: mudança de dados + mudança de componente.
- 🟡 **Sem validação de schema:** erro de digitação em `jobs.json` (ex: `"ongoing": "true"` em vez de `true`) causa erro silencioso ou crash em runtime sem mensagem útil. Considerar `zod` ou `ajv` para validação no build.
- 🟡 **Campo `images[]` potencialmente morto:** presente no schema de projetos mas sem referência encontrada em `Projects.tsx`. Pode ser legado de uma feature de galeria nunca implementada.
- 🟡 **Crescimento manual:** adicionar um novo job ou projeto requer edição manual do JSON e deploy. Para um portfólio pessoal, é aceitável; para escala maior, um CMS headless seria apropriado.
